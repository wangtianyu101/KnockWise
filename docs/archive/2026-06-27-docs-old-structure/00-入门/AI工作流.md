# AI Coding 工作流（Intervue）

> **一页纸讲清我们怎么跟 AI 协作干活。**
> 适用对象：团队成员、面试官、未来的自己。

---

## 一句话定位

> **AI 是干活的，不是决策的。**
> 我们让 AI 做体力活（写代码、查 API、生成 schema），
> **决策 + 验证 + 通过永远是人**。

---

## 为什么需要这个工作流

长期高频用 AI coding 会撞上 5 个真实问题：

| 问题 | 表现 | 我们的解 |
|---|---|---|
| ① 不理解意图 | AI 给的不是你要的 | 文档先行 + 多次确认 |
| ② 代码先行 | 不等"通过"就开始写 | 6 步流程 + DOD 清单 |
| ③ 上下文超限 | 单次任务太大 | 按 phase 拆分 |
| ④ 多 agent 冲突 | 多个 AI 各做各的 | 必读 `目前缺陷.md` + git status |
| ⑤ 没有验证 | AI 写完不知道对错 | 单测强制 + pre-commit hook |

完整问题分析见 `obsidian/coding/AI代码工具使用心得/AI工具与流程体系化分析.md`

## 6 步流程（不可跳级）

```
[0] 调研 → [1] 设计 → [2] 验证 → [3] 详细化 → [4] 页面 → [5] 通过 → [6] 实施
       "调研"   "做设计"   "验证"     "细化"     "出页面"   "通过"   "开始实施"
```

**触发词**：每个步骤必须由对应关键词触发，AI 不擅自进下一步。

| 步 | 触发词 | AI 要做 |
|---|---|---|
| 0 | "调研" | 读 `目前缺陷.md` + `git log -20` + 相关议題，出一页纸 |
| 1 | "做设计" | 写设计文档：目标、范围、**全局架构图**、用户旅程 |
| 2 | "验证" | 自查：完整性 / 可行性 / 一致性 / 冲突点 |
| 3 | "细化" | 加 API/表/service/错误码/迁移 SQL |
| 4 | "出页面" | ASCII 草图 + 流程图 + 组件清单 |
| 5 | "通过" | 你明确批准 → 才进 6 |
| 6 | "开始实施" | 按阶段 3 落代码 + 写测试 |

**完整定义在** `CLAUDE.md`（项目根目录）

## 分支流程（非新功能也要走）

| 类型 | 路径 | 关键约束 |
|---|---|---|
| 新功能 | 0→1→2→3→4→5→6 | 完整流程 |
| Bug 修复 | 登记 → fix commit → 回归测试 | 必须有测试 |
| 重构 | 登记议題 → commit 引用议題号 | 不改业务行为 |
| 议題关闭 | 直接执行调研结论 | commit 标题含议題号 |

---

## 三条硬规则（缺一不可）

### 规则 1：单测强制（CLAUDE.md 第六节）

- 新 service / 新 endpoint / 新 schema → **必须有测试**
- Bug 修复 → **必须有回归测试**
- `pytest` / `vitest` 全绿才 commit

### 规则 2：pre-commit hook

- 装好后 `git commit` 自动跑 `tsc` + `pytest`
- 任一失败 → **禁止 commit**
- 紧急跳过：`PRE_COMMIT_SKIP=1 git commit ...`

### 规则 3：阶段 6 DOD（完成定义）

实施完必须 6 项全过：
- [ ] 所有 phase 的 doc 已 commit
- [ ] `目前缺陷.md` 议題状态更新（📋→🚧→✅）
- [ ] pre-commit hook 通过
- [ ] **核心 service 覆盖率 ≥ 80%**
- [ ] 没有"未通过就实施"
- [ ] 用户口头确认

**核心 service 清单**（6 个）：
`interview_service` / `learning_progress_service` / `question_bank_service` /
`qa_service` / `recommendations_service` / `study_plan_service`

## 验证手段（不止单测）

| 手段 | 何时用 |
|---|---|
| **单元测试** | 每个新函数、每个 bug 修复 |
| **tsc 类型检查** | 前端每个 commit |
| **手动 git diff** | 合并前 review AI 改了什么 |
| **运行日志** | 集成测试（生产前要加 trace） |
| **Code Review** | 多人协作时强制 |

> ⚠️ **AI 的 happy path 你不能信。** 必须自己跑一遍 + 看 diff + 看产物，不能只看 commit message。

## AI 该做什么 / 不该做什么

### ✅ 让 AI 做
- 写代码（CRUD、schema、endpoint）
- 查 API（看文档、找函数签名）
- 写测试（基于已存在的行为）
- 写文档（从代码生成设计文档初稿）
- 重构（按议題清理）
- 跑命令（git 操作、依赖安装）

### ❌ AI 不做
- 决定"做什么"（产品决策归人）
- 决定"做到什么程度算通过"（归人）
- 合并到 main（必须人工）
- 删/改 `seed_data/*.json`（CLAUDE.md 黑名单）
- 删/改 `.env.local`、`.venv/`、`node_modules/`（同上）
- 删 MySQL 真实数据（同上）

## 一日工作流（实操版）

```
早上开工：
  1. git pull
  2. cat docs/40-追踪/目前缺陷.md  ← 看议題
  3. 跟 AI 说"调研：要做 XXX"

接到新需求：
  0 调研 → 1 设计 → 2 验证 → 3 细化 → 4 页面 → 5 通过 → 6 实施
  每步必等你的明确指令

写代码时：
  AI 写完 → 你看 diff → 跑测试 → commit
  pre-commit hook 自动兜底

遇到 bug：
  1. 先复现
  2. 在 目前缺陷.md 加一条
  3. 让 AI 修 + 写回归测试
  4. commit 标题：fix(模块): 描述

遇到重构：
  1. 在 目前缺陷.md 加议題
  2. 让 AI 调研 → 你看方案 → 批准 → AI 实施
```

## 失败案例（避免重蹈）

❌ **AI 不等"通过"就写代码**
→ 用 6 步流程 + "实施前必说'开始实施'"硬约束

❌ **AI 一次写 22 个 endpoint**
→ 单次 commit 不超 5 个新文件；超过 → 拆分

❌ **AI 写完不知道对错**
→ 单测 + pre-commit hook + 你手动跑一次

❌ **多 agent 互相冲突**
→ 开工前必读 `目前缺陷.md` + `git status`

❌ **议題堆积无人推进**
→ 调研步骤强制读 `目前缺陷.md`

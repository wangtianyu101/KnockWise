## 1. 问题定义

### 用户痛点
<!-- TODO: 人填 — 你自己用 6 步流程时，最痛的点是什么？ -->

- AI 协作流程（SOP）目前是**个人隐性知识** — 只在用户自己脑子里，团队 / 朋友 / 同事无法复用
- Claude Code / Codex / Cursor 等 agent **没有统一接入** 6 步流程的协议层，每次都要重新教 AI
- 工作流跑挂 / 跑偏 / 中断时**没有可观测**，不知道哪一步出问题
- 新增能力（如"加一个 PDF 简历解析"）需要**改 AI prompt + 改主代码**，无法热插拔
- 6 步流程的 9 份文档**没有标准化产出**（哪些必填、哪些可选、什么格式）

### 时机
- 2026 上半年 AI agent 框架爆发（LangGraph 1.x / Claude Agent SDK / OpenAI Agents SDK / MCP 协议成事实标准）
- 百度 / Comate / Cursor 都在做"AI coding 工作流 + 动态组件"
- 项目自身的 V2 沉淀层（6 services + 6 endpoints）已跑通，**有真实业务负载可承载**
- 用户**刚从百度面试回来**，对这套架构有体感，是立项黄金窗口

### 不做会怎样
<!-- TODO: 人填 — 你最担心什么？ -->

- 个人 SOP 继续留在脑子里，**换项目 / 换人 = 全部重做**
- AI agent 接入协议五花八门，**兼容性债务**越积越重
- 错过 2026 上半年的 agent 框架窗口期，**下半年技术债翻倍**
- **简历上少一个能讲清楚的"产品级"项目**（vs 把工程化做完整可讲 1 小时）

## 2. 目标用户

### 角色
<!-- TODO: 人填 — 第一波用户是谁？ -->

**主用户**（必填）：
- **独立开发者**（用户本人） — 用 6 步流程 + 多种 AI agent 协作写代码
- AI 工具**重度用户**（≥ 1 小时/天用 Claude Code / Cursor / Codex 等）
- 重视**质量可追溯**（每步产物能 commit / 能复盘）

**次用户**（可选）：
- 小团队 lead（2-5 人） — 想把团队的"AI 协作 SOP"标准化
- AI 编程训练营讲师 — 想给学员一套"人 + AI 协作"的教学模板

### 场景
- **场景 1**（主）：本地起新功能，按"调研→规格→计划→拆分→实现→验证→复盘"跑 6 步
- **场景 2**：在 Claude Code 里写代码，AI 主动调用 MCP 工具走 6 步
- **场景 3**：多 agent 协作（Claude Code + Codex 同时跑），统一走同一个工作流引擎
- **场景 4**：复盘后，沉淀的经验自动更新 CLAUDE.md / DOD.md，下次直接用

### 频率
- 场景 1：**每天 1-3 次**（新功能 / 修 bug / 重构）
- 场景 2：**每周 5-10 次**
- 场景 3：**每月 1-2 次**（复杂任务）
- 场景 4：**每周 1 次**（每周末复盘）

## 3. 价值主张

### 用户价值
- **省心**：AI 自动按 6 步走，不用每次重新 prompt
- **可追溯**：每步产物 commit + 状态可查，**挂了一键恢复**
- **可扩展**：加新能力 = 写一个 Component 类，**不碰主程序**
- **跨 agent 兼容**：Claude / Codex / Cursor 都能用同一套工作流

### 商业价值（独立开发者角度）
<!-- TODO: 人填 — 这块你想要什么？ -->

- **简历价值**：1 个月做完整 = 1 个能讲 1 小时的产品级项目（工作流引擎 + 动态组件 + MCP 协议）
- **内容价值**：可发博客 / 视频，主题"AI coding 工作流" 2026 是热点
- **复用价值**：下一个项目直接 `pip install` 复用，**不重造轮子**
- **开源价值**（可选）：发 GitHub，攒 star（前提是文档全 + 演示好）

## 4. MVP 范围（关键）

### 包含（最小可用版本必须有的）

| # | 功能 | 验收标准 | 估时 |
|---|---|---|---|
| 1 | **6 步状态机引擎** | start → advance × 7 → completed 跑通，SQLite 持久化 | 0.5 天 |
| 2 | **Pydantic 组件协议** | 写一个 Component 类就能被注册 + 执行 | 0.5 天 |
| 3 | **MCP Server 骨架** | Claude Code 能调 `start_research` / `submit_spec` 等 7 个工具 | 1 天 |
| 4 | **Claude Code Skill** | SKILL.md 描述 6 步流程触发条件，AI 自动识别 | 0.5 天 |
| 5 | **1 个真实 Component 跑通** | 用 `obsidian_service` 做 `0_research` 的真实组件，**端到端 demo** | 1 天 |
| 6 | **可观测性基础** | structlog JSON 日志 + run_id 关联 | 0.5 天 |
| 7 | **单测覆盖 ≥ 80%** | 核心 engine + ComponentRegistry + 1 个示例 component | 0.5 天 |

**L1-L2 周期：1 周**（40h，5 个工作日）

### 不包含（明确不做的）

| # | 不做的 | 理由 |
|---|---|---|
| 1 | **PyPI 发布** | 1 人项目，包发布 + 文档站维护成本高；内置到 Intervue 仓库更轻 |
| 2 | **可视化拖拽 UI**（vs Dify/Coze）| L4 SaaS 复杂度，**MVP 不做**；CLI + MCP 协议就够 |
| 3 | **插件市场**（公开注册中心）| 单人项目无第三方开发者；私有组件库即可 |
| 4 | **远程 MCP transport**（SSE/HTTP）| 远程场景有但少；本地 stdio 先跑通 |
| 5 | **多租户 / 团队协作** | 个人项目先做精，**v2 再做团队化** |
| 6 | **Web Dashboard** | CLI 状态 + 日志够用；**Web UI 是 v2** |
| 7 | **AI 自动选工作流**（让 AI 拼步骤）| MVP 走固定 6 步，AI 拼工作流是 v3 |

### 未来迭代（v2+ 可考虑）

| 版本 | 内容 |
|---|---|
| **v2** | 远程 MCP（SSE）+ Web Dashboard（看 run 状态）+ 多租户支持 |
| **v3** | 插件市场 + 第三方组件 + 社区文档 |
| **v4** | AI 自动拼工作流（ReAct over workflow） + 可视化编辑 |

## 5. 成功指标

### 核心指标（1-3 个量化）

| # | 指标 | 数字 | 衡量方式 |
|---|---|---|---|
| 1 | **真实业务跑通** | 至少 **1 个真实 V2 沉淀层组件**（如 `obsidian_sediment_service`）**走完 6 步 + 产出 commit** | git log + verify.md |
| 2 | **单测覆盖** | 核心 engine + registry + 1 个示例 component **≥ 80%** | pytest --cov |
| 3 | **可演示** | 录制 **1 段 5 分钟端到端 demo 视频**（Claude Code 调 MCP → 6 步 → commit）| 录屏文件 |

### 监控方式
- 指标 1：verify.md + git log（自动统计）
- 指标 2：pytest --cov 输出
- 指标 3：录屏文件 + 博客文章

### 合格线（什么数字算"成功"）
- **L1-L2 上线 1 周后**：
  - 指标 1：1 个 ✅
  - 指标 2：≥ 80% ✅
  - 指标 3：1 段视频 ✅
  - **3/3 达 = L1-L2 成功**，可继续做 v2

### 防失败线
- L2 上线 **2 周后**仍无真实组件接入 → **回 0 步调研，重新评估需求**
- pytest 覆盖 < 60% → **必须补测，不进 v2**
- demo 视频录不出来（端到端跑挂）→ **先 fix bug，再录**
